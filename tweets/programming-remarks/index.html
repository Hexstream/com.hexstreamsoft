Documentation (6) ⚓

External documentation: explain things comprehensively, organize topics logically. Autogen'd doc: explains the what but not the how or why.

— Jean-PhilippeParadis (@HexstreamSoft) July 28, 2012

Strong documentation allows you to get away with novel approaches, otherwise your insightful code just looks "unnecessarily complicated".

— Jean-PhilippeParadis (@HexstreamSoft) July 28, 2012

Attempting to document things rather formally and comprehensively can reveal accidental complexity: ameliorate design for easier explaining.

— Jean-PhilippeParadis (@HexstreamSoft) July 28, 2012

Another reason to prefer external documentation: Hate to clutter code with comments. Mixing code and documentation is like mixing two wines!

— Jean-PhilippeParadis (@HexstreamSoft) July 28, 2012

Documentation: Copy CLHS style of assuming multiple implementations. Successive versions ~= different impls that just happen to be similar.

— Jean-PhilippeParadis (@HexstreamSoft) July 29, 2012

Writing code makes you feel productive. Writing documentation makes your users feel productive. Don't short-change your users. Document well

— Rob Pike (@rob_pike) June 26, 2012
